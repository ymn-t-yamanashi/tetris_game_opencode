# コーディング規約

このリポジトリで遵守すべきコードスタイルとベストプラクティスをまとめます。

## 1. ファイル構成
- `lib/` 以下にビジネスロジック、`web/` 以下に Phoenix のコードを配置します。
- テストは `test/` に同名ディレクトリで保持し、ファイル名は `*_test.exs` 形式とします。

## 2. コーディングスタイル
- Elixir は **マジックナンバー** を避け、定数化します。例: `@max_retry = 3`。
- 名前はスネークケースで記述し、関数名は動詞＋目的語の形にします。
- コメントは日本語で書き、必要な場所のみ簡潔に説明を入れます。

## 3. Phoenix 標準規約
- LiveView テンプレートは `<Layouts.app>` でラップし、`flash` を渡します。
- コンポーネントは `core_components.ex` の `<.input>`, `<.icon>` 等を使用してください。
- ルーティングはスコープ内にまとめ、認証が必要なパスは `live_session :auth, ...` に入れます。

## 4. HEEx テンプレート規約
- HEEx ファイルは `.html.heex` 拡張子を使用します。
- タグの属性はリスト構文で記述し、複数クラスは `["class1", "class2"]` の形にします。
- `<% ... %>` ブロックはタグ内でのみ使用し、属性値には `{...}` で挿入します。
- コメントは `<%!-- comment --%>` を使い、インラインコメントは `phx-no-curly-interpolation` 属性を付与した `<code>` タグで行います。

## 4. テスト方針
- ユニットテストは関数単位で書き、外部依存はモック化します。
- LiveView のテストは `Phoenix.LiveViewTest` を利用し、DOM 要素の存在を確認します。
- カバレッジは 80% 以上を目指してください。

## 5. コミットメッセージ
- 「タイプ: 概要」形式で書きます。例: `feat: ユーザー登録機能追加`。
- 変更内容が明確に分かるように、何をしたかを簡潔に記述します。
